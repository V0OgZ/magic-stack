              20 = move from L to BR
              21 = move from TL to B
              22 = move from T to BL
              23 = move from TR to L
              24 = move from R to TL

                            +-+-+-+
                TL--T--TR   | |D| |   (D)estination
                |       |   +-+^+-+
                L       R   | |m| |   <- though this cell move is from BL to T
                |       |   +-/-+-+
                BL--B--BR   |H| | |   (H)ero
                            +-+-+-+

              It is easy to see that moves happen from one of 8 points to one
              of 3 points on the opposite side of the square: TR to BL/B/BR,
              L to TR/R/BR, etc. - hence ADAG has 8*3=24 variations in total.
              But this doesn't cover all combinations: 8*8 = 64 minus 8 clearly
              meaningless (to self, e.g. T to T) = 56. This includes 8*2 (to
              adjacent: L to TL) and 8 (diagonal: T to L) = 24, of which some
              are not meaningless because SoD disallows entering actionable
              spot from top or leaving it to top so seemingly irrational detours
              (o) are required when moving D to H (TL to L) or H to D as below.

                +-+-+-+     move from T to/from TL = meaningless
                |D| | |     move from T to/from TR = meaningless
                +-\-+-+     move from R to TR = 18
                |H-o| |     move from TR to R = 3
                +-+-+-+     move from L to TL = 8
                | | | |     move from TL to L = 23
                +-+-+-+     move from B to/from BL = meaningless
                adjacent    move from B to/from BR = meaningless
                            move from R to/from BR = meaningless
                            move from L to/from BL = meaningless

                +-+-+-+     move from T to L = 23
                | |H| |     move from L to T = 1
                +-+v+-+     move from T to R = 3
                |D<o| |     move from R to T = 17
                +-+-+-+     move from B to/from L = meaningless
                | | | |     move from B to/from R = meaningless
                +-+-+-+
                diagonal

                This makes for 24+8*2+8 = 48; remaining 64-8 = 8 are:

                +-+-+-+     move from BL to TL = 8
                |D| | |     move from BR to TR = 18
                +-\-+-+     move from TL to BL = 22
                | |o| |     move from TR to BR = 4
                +-/-+-+     move from BR to BL = 6
                |H| | |     move from TR to TL = 24
                +-+-+-+     move from BL to BR = 20
                BL to TL    move from TL to TR = 2

TSHRE         ADVMAP's shroud cornering visible tiles in various combinations

              According to my in-game tests with SoD, visual inspection of
              TSHRE frames and intuition, an invisible tile's frame depends on
              visibility of 8 adjacent tiles. It doesn't depend on the tile
              itself since no matter of the sides' visibility, if the center
              is visible then the tile is shown without shroud.

              Given the above, there are 2 ^ 8 = 256 combinations. However,
              TSHRE has much fewer frames. I formed a hypothesis that
              visibility of a corner is irrelevant if any adjacent side is
              visible (for example, if a frame shows Top as visible then it
              handles Top, Top/Top-Left, Top/Top-Right, Top/Top-Left/Top-Right).
              This gives 47 combinations.

              1. Most frames are used for specific combinations, either
                 verbatim (e.g. 06) or horizontally flipped (e.g. 22).
              2. 4 combinations have multiple frames (same images); which
                 one is used depends on coordinates of the tile being concealed
                 (e.g. 04/05) - 0-based, from top-left corner.
              3. Other 4 combinations having multiple frames select one by
                 states of corners (even in presence of a visible side) - if
                 any corner is visible, one frame is used (e.g. 12) and if none
                 are, another is used (14). These are marked with the
                 disjunction operator (V).
              4. When checking state of tile (_) outside of map boundaries, it
                 +--_--+  seems fit to assume state of the adjacent border tile
                 |  T  |  (T). For corners (+), always assume "not visible".
